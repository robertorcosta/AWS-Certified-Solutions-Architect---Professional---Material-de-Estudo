Domínio 1: Soluções para Complexidade Organizacional (26%)

Este domínio avalia sua capacidade de projetar arquiteturas que funcionam em ambientes corporativos complexos com múltiplas contas, conectividade híbrida e governança em escala.

1.1 Estratégias de Conectividade de Rede

AWS Transit Gateway

Hub central para conectar múltiplas VPCs e redes on-premises. Substitui o modelo de peering mesh complexo.

  • Suporta: peering entre Regions, roteamento entre VPCs, integração com Direct Connect e VPN
  • Route Tables: permite segmentação de tráfego (ex: isolar VPCs de produção de desenvolvimento)
  • Multicast: suportado nativamente
  • Bandwidth: até 50 Gbps por attachment
  • Inter-Region Peering: tráfego criptografado entre TGWs de diferentes regiões

Cenário comum na prova: "Empresa com 50+ VPCs precisa de conectividade full-mesh com roteamento centralizado"

Documentação Transit Gateway
AWS Direct Connect

Conexão dedicada entre seu data center e a AWS. Reduz custos de rede, aumenta largura de banda e oferece latência mais consistente.

  • Tipos: Dedicated (1/10/100 Gbps) e Hosted (50 Mbps a 10 Gbps)
  • Virtual Interfaces:
    • Private VIF — acessa VPCs (via Direct Connect Gateway para múltiplas VPCs/Regions)
    • Public VIF — acessa serviços públicos da AWS
    • Transit VIF — conecta ao Transit Gateway
  • LAG (Link Aggregation Group): agrupa múltiplas conexões para aumentar throughput
  • Resiliência: para alta disponibilidade, use 2 conexões em locais diferentes + VPN como fallback
  • MACsec: criptografia na camada 2 (disponível em 10/100 Gbps dedicated)

Cenário comum: "Migração de grandes volumes de dados com latência consistente" ou "Conexão híbrida resiliente com failover"

Documentação Direct Connect
AWS Site-to-Site VPN

Conexão criptografada (IPsec) entre rede on-premises e VPC via internet pública.

  • Virtual Private Gateway (VGW): endpoint do lado AWS para uma única VPC
  • Transit Gateway + VPN: permite VPN para múltiplas VPCs
  • Accelerated VPN: usa AWS Global Accelerator para melhorar performance
  • Throughput: ~1.25 Gbps por túnel (2 túneis por conexão VPN)
  • ECMP: Equal-Cost Multi-Path — com Transit Gateway, permite agregar múltiplas VPNs para mais throughput

Cenário comum: "Precisa de conexão rápida enquanto Direct Connect é provisionado" (VPN como backup ou solução temporária)

DNS Híbrido (Route 53 Resolver)

Permite resolução de DNS bidirecional entre rede on-premises e VPCs.

  • Inbound Endpoint: permite que o DNS on-premises resolva nomes no Route 53 (on-prem → AWS)
  • Outbound Endpoint: permite que VPCs resolvam nomes no DNS on-premises (AWS → on-prem)
  • Resolver Rules: regras de encaminhamento condicional (ex: "*.corp.local" → DNS on-prem)
  • RAM Sharing: compartilhe regras entre contas via Resource Access Manager

Cenário comum: "Resolver nomes de domínio privados entre AWS e on-premises em ambiente multi-account"

1.2 Controles de Segurança

IAM Avançado e Cross-Account Access

Métodos de Acesso Cross-Account

  • IAM Roles (AssumeRole): método preferido. Conta A assume role na Conta B. Temporário e auditável.
  • Resource-based Policies: S3 bucket policy, KMS key policy, SQS policy — permite acesso direto sem assumir role.
  • AWS Organizations SCPs: limites máximos de permissão (guardrails) em nível de OU/conta.
  • Permission Boundaries: limite máximo para roles/users dentro de uma conta.

Ordem de Avaliação de Políticas

  1. SCP (Organization) — deny explícito prevalece
  2. Permission Boundary — limita o teto
  3. Identity-based Policy — o que o user/role pode fazer
  4. Resource-based Policy — o que o recurso permite
  5. Session Policy — restrição adicional em sessões assumidas
🔑 Conceito-chave para a prova Um deny explícito em QUALQUER nível sempre vence. Para cross-account, AMBOS os lados precisam permitir (exceto resource-based policies do S3 que podem conceder direto).
Lógica de Avaliação IAM Permission Boundaries
AWS IAM Identity Center (SSO)

Gerenciamento centralizado de acesso a múltiplas contas AWS e aplicações de negócio. Substitui o antigo AWS Single Sign-On.

  • Integração: Active Directory (AD Connector ou AWS Managed AD), provedores SAML 2.0 externos (Okta, Azure AD), ou Identity Store built-in
  • Permission Sets: coleção de IAM Policies atribuída a users/groups para definir acesso AWS por conta
  • ABAC: Attribute-Based Access Control — define permissões uma vez, modifica acesso mudando atributos (cost center, title, locale)
  • Multi-Account: um login único para acessar qualquer conta da Organization
  • Application Assignments: SSO para apps SAML 2.0 (Salesforce, Box, Microsoft 365)
  • EC2 Windows: login único em instâncias Windows via SSO

Cenário comum: "Centralizar autenticação para 100+ contas AWS com identidades corporativas existentes"

💡 SAML 2.0 Federation vs IAM Identity Center SAML 2.0 Federation é o "jeito antigo" — Identity Center é o recomendado. Se a questão menciona setup novo com AD, a resposta é IAM Identity Center. Se menciona sistema legado já com SAML configurado, pode ser SAML direto.

Criptografia e Gerenciamento de Chaves

AWS KMS (Key Management Service)

Sempre que ouvir "criptografia" para um serviço AWS, provavelmente envolve KMS. Keys são regionais.

Tipos de KMS Keys

TipoGerenciamentoRotaçãoVisibilidade
Customer Managed Key Você cria, gerencia, habilita/desabilita 90-2560 dias (default 365) + on-demand Total (policy + CloudTrail)
AWS Managed Key AWS gerencia (aws/s3, aws/ebs...) Obrigatória a cada 1 ano Pode ver policy e auditar
AWS Owned Key AWS usa internamente Varia Nenhuma (não pode ver/auditar)

Key Types

  • Symmetric (AES-256): uma chave para encrypt/decrypt. Usada por todos os serviços AWS integrados. Necessária para envelope encryption. Nunca exposta sem criptografia.
  • Asymmetric (RSA/ECC): par público/privado. Public key é downloadable. Use case: encrypt fora da AWS por quem não pode chamar a KMS API.

Key Material Origin

  • KMS (AWS_KMS): padrão, AWS gera e gerencia
  • External (EXTERNAL): BYOK — você importa. Responsável pela segurança fora da AWS. Sem rotação automática.
  • Custom Key Store (AWS_CLOUDHSM): material armazenado em cluster CloudHSM que você gerencia. Precisa de pelo menos 2 HSMs ativos.

Multi-Region Keys

  • Mesmo key ID, key material e rotação em todas as regiões
  • Encrypt em uma região, decrypt em outra (sem re-encrypt ou API cross-region)
  • NÃO são globais — são Primary + Replicas gerenciadas independentemente
  • Use cases: DR, DynamoDB Global Tables, Active-Active multi-region, distributed signing

Cross-Account KMS

Key Policy permite conta externa → Role na conta externa usa a chave. Ambos os lados precisam permitir.

AWS CloudHSM

  • Hardware dedicado (FIPS 140-2 Level 3)
  • Você gerencia as chaves (AWS não tem acesso)
  • Use quando: compliance requer HSM dedicado, SSL offloading, Oracle TDE
  • Integra com KMS via Custom Key Store

Envelope Encryption

Data Key criptografa os dados → CMK criptografa a Data Key. Padrão usado pelo SDK da AWS. Necessário para arquivos > 4KB.

CloudTrail — Detalhes importantes

  • Tipos de eventos:
    • Management Events: operações em recursos (habilitado por padrão)
    • Data Events: operações em dados — S3 GetObject/PutObject, Lambda Invoke (NÃO habilitado por padrão, alto volume)
    • Insights Events: detecta atividade incomum (bursts de IAM, picos de API)
  • Retenção: 90 dias no CloudTrail → para reter mais, envie para S3 + consulte com Athena
  • Organization Trail: trail único na management account captura logs de TODAS as contas-membro automaticamente
  • Log File Integrity: validação SHA-256 para hashing e signing
  • Tempo de entrega: até 15 minutos para entregar eventos

Reagir a eventos CloudTrail (do mais rápido ao mais lento):

  1. EventBridge: qualquer API call pode triggerar (mais rápido e reativo)
  2. CloudWatch Logs: stream de eventos com metric filters para anomalias
  3. S3 Delivery: a cada 5 minutos (análise de integridade, cross-account, long-term)

CloudTrail Solution Architectures (comuns na prova)

  • Multi-Account Logging: S3 bucket centralizado com bucket policy para cross-account delivery
  • Alertas: CloudTrail → CloudWatch Logs → Metric Filter (ex: count TerminateInstances) → Alarm → SNS
  • S3 Security: Versioning + MFA Delete + Object Lock + SSE-KMS + Lifecycle para Glacier

Serviços-chave para DR

  • AWS Elastic Disaster Recovery (DRS): replicação contínua block-level de servidores para região de DR. Agent-based. Failover em minutos. Suporta failback.
  • S3 Cross-Region Replication: replica objetos automaticamente entre regiões
  • RDS Multi-AZ: failover automático para standby (síncrono, 30-60s)
  • RDS Cross-Region Read Replicas: replicação assíncrona para outra região (pode ser promovida)
  • Aurora Global Database: replicação < 1 segundo entre regiões, failover em < 1 minuto
  • DynamoDB Global Tables: multi-region, multi-active com replicação automática

DR Tips (do PDF do Stephane Maarek)

  • Backup: EBS Snapshots, RDS automated backups, S3 + Lifecycle + CRR, Storage Gateway, Snowball
  • HA: Route 53 failover, RDS Multi-AZ, ElastiCache Multi-AZ, EFS, S3. VPN como recovery do Direct Connect.
  • Replication: RDS Cross-Region, Aurora Global DB, DB replication on-prem → RDS, Storage Gateway
  • Automation: CloudFormation/Beanstalk para recriar ambientes, CloudWatch Alarms + recovery actions, Lambda automations
  • Chaos Engineering: AWS Fault Injection Simulator (FIS) — injeta falhas (CPU spike, terminate, network) em EC2, ECS, EKS, RDS. Usa templates pré-construídos.

AWS Backup

  • Gerenciamento centralizado de backups cross-service (EC2, EBS, S3, RDS, Aurora, DynamoDB, EFS, FSx, DocumentDB, Neptune, Storage Gateway)
  • Cross-region e cross-account backups
  • PITR (point-in-time recovery) para serviços suportados
  • Tag-based backup policies, scheduling (cron), retention, transition to cold storage
  • Backup Vault Lock (WORM): backups não podem ser deletados — nem pelo root user! Proteção contra ransomware.

Alta Disponibilidade e Auto Scaling

Padrões de Alta Disponibilidade

  • Multi-AZ: mínimo para produção. Protege contra falha de um datacenter.
  • Multi-Region: protege contra falha regional. Mais complexo e caro.
  • ELB Cross-Zone: distribui tráfego uniformemente entre AZs
  • Auto Scaling Groups: min/max/desired capacity. Políticas: Target Tracking, Step, Scheduled, Predictive.

Route 53 Routing Policies para HA

  • Failover: ativo/passivo com health checks
  • Latency-based: roteia para a região com menor latência
  • Weighted: distribui % do tráfego (blue/green, canary)
  • Geolocation/Geoproximity: roteia por localização do usuário
  • Multivalue Answer: retorna múltiplos IPs saudáveis

1.4 Ambiente Multi-Account

AWS Organizations

Gerenciamento centralizado de múltiplas contas AWS em uma hierarquia de OUs (Organizational Units).

  • Management Account: conta raiz que cria a organização. NÃO use para workloads.
  • SCPs (Service Control Policies): guardrails que limitam o que contas/OUs PODEM fazer. Não concedem permissões, apenas restringem.
  • Consolidated Billing: uma fatura única, descontos por volume compartilhados
  • Delegated Admin: delega administração de serviços (Security Hub, GuardDuty) para conta membro

Estrutura Recomendada de OUs

  • Security OU: Log Archive + Security Tooling
  • Infrastructure OU: Networking, Shared Services
  • Workloads OU: Production, Staging, Development
  • Sandbox OU: contas de experimentação com restrições
AWS Organizations SCPs
AWS Control Tower

Orquestra a criação e governança de um ambiente multi-account seguro (Landing Zone).

  • Landing Zone: ambiente pré-configurado com melhores práticas
  • Guardrails (Controls):
    • Preventive — SCPs que bloqueiam ações (ex: bloquear criação de access keys para root)
    • Detective — AWS Config Rules que detectam não-conformidade (ex: detectar se MFA do root está habilitado)
    • Proactive — CloudFormation hooks que validam antes do deploy
  • Guardrail Levels:
    • Mandatory: habilitados automaticamente (ex: bloquear leitura pública no Log Archive)
    • Strongly Recommended: best practices opcionais (ex: criptografia EBS)
    • Elective: comuns em enterprise (ex: bloquear delete sem MFA em S3)
  • Account Factory: vending machine para criar novas contas padronizadas usando AWS Service Catalog
  • Customizations for Control Tower (CfCT): aplica templates CloudFormation automaticamente em novas contas

Detect and Remediate Policy Violations

Fluxo: Control Tower detecta non-compliance → SNS → EventBridge → Lambda executa remediação automática

Cenário comum: "Precisa provisionar novas contas automaticamente com baseline de segurança"

Tipos de Policies além de SCPs

Detalhes importantes para a prova

  • SCPs não afetam Management Account — nunca restringem a conta de gerenciamento
  • SCPs não afetam Service-Linked Roles — roles criadas por serviços AWS para integrações
  • SCP precisa de Allow explícito desde a root até o alvo na hierarquia
  • Reserved Instances são compartilhadas na organização (consolidated billing), mas pode desligar sharing por conta
  • OrganizationAccountAccessRole: role com full admin criada automaticamente em novas contas (deve ser criada manualmente se conta é convidada)
  • Restringir criação sem Tags: SCP pode bloquear ec2:RunInstances se não tiver tags específicas (usando aws:TagKeys)
  • Restringir por Região: SCP com condição aws:RequestedRegion para bloquear regiões não autorizadas

Infrastructure as Code (IaC)

AWS CloudFormation

  • Stacks: unidade de deployment. Pode ter dependências entre stacks (cross-stack references via Export/Import)
  • StackSets: deploy em múltiplas contas e regiões simultaneamente

EBS — Detalhes do Exame (do PDF)

  • Volume Types:
    • gp2/gp3 (SSD): uso geral, boot volumes. gp3: 3000 IOPS baseline, configurável até 16.000
    • io1/io2 Block Express: mission-critical, até 256.000 IOPS. Suporta Multi-Attach na mesma AZ.
    • st1 (HDD): throughput-intensive, frequentemente acessado (big data, log processing)
    • sc1 (HDD): lowest cost, infrequente (cold data, archive)
  • Snapshots: incrementais, armazenados no S3. FSR (Fast Snapshot Restore) para evitar latência de primeiro acesso.
  • Encryption: account-level setting por região para criptografar automaticamente novos volumes
  • Multi-Attach (io1/io2): mesmo volume em múltiplas EC2 na mesma AZ. Requer cluster-aware filesystem.
  • Data Lifecycle Manager: automatiza criação/retenção/deleção de snapshots por tags

Instance Store vs EBS

  • Instance Store: fisicamente no servidor, efêmero, milhões de IOPS, perdido no stop/terminate
  • EBS: network drive, persistente, até 256K IOPS (io2 Block Express), sobrevive reboot

EFS — Detalhes do Exame (do PDF)

  • Managed NFS, multi-AZ, Linux only (POSIX), pay-per-use, escala para petabytes
  • Performance Modes: General Purpose (default, latência) vs Max I/O (throughput, paralelo)
  • Throughput Modes: Bursting | Provisioned | Elastic (auto-scale, até 3 GiB/s reads)
  • Storage Classes: Standard → Infrequent Access → Archive (50% mais barato). Lifecycle policies.
  • One Zone: 90%+ economia, bom para dev. Compatível com IA.
  • Access Points: restringem acesso a diretórios específicos com POSIX user/group enforcement
  • Cross-Region Replication: RPO/RTO de minutos para DR
  • On-premises access: via Direct Connect ou VPN (mount por IPv4, não DNS)

FSx — Detalhes do Exame (do PDF)

  • FSx for Windows: SMB + NTFS + AD + DFS Namespaces. Multi-AZ. Backup daily para S3. Acessível via DX/VPN.
  • FSx for Lustre:
    • Scratch FS: temporário, sem replicação, burst 6x (short-term processing, cost optimized)
    • Persistent FS: replicado na mesma AZ, self-healing (long-term, sensitive data)
    • Data Lazy Loading: só carrega do S3 os dados que realmente processa (reduz custo e latência)
  • FSx for NetApp ONTAP: NFS + SMB + iSCSI, shrink/grow automático, snapshots, point-in-time cloning
  • Migrar Single-AZ → Multi-AZ: usar DataSync ou backup/restore
  • Reduzir volume size: criar novo FSx menor + DataSync para sync

Storage Gateway Types

  • S3 File Gateway: NFS/SMB → S3 (Standard, IA, One Zone-IA, Intelligent-Tiering). Cache local. AD integration para SMB.
  • FSx File Gateway: cache local de FSx for Windows (para branches com latência)
  • Volume Gateway: iSCSI → EBS snapshots. Cached (low latency recent data) vs Stored (full dataset on-prem, scheduled backups)
  • Tape Gateway: VTL via iSCSI para S3 + Glacier (substitui fitas físicas)
💡 File Gateway: Object Lock + Versioning Habilita WORM data. Modificações criam nova versão sem afetar as anteriores. Use "RefreshCache" API para ser notificado de restores.

Fargate vs EC2 Launch Type

  • Fargate: serverless, paga por vCPU/memória/segundo. Ideal quando não quer gerenciar infra.
  • EC2: mais controle, GPU support, preços melhores com Spot/RI para workloads previsíveis.
  • Fargate Spot: tasks que podem ser reclamadas pela AWS (70% desconto). Especifique baseline on-demand + extra em Spot.

ECS — Conceitos-chave (do PDF)

  • Task Definition: JSON que define container image, CPU, RAM, ports, IAM role
  • ECS Task: instância em execução de uma Task Definition
  • ECS Service: define quantas tasks rodar e como (desired count, deployment strategy)
  • EC2 Instance Profile: role da instância EC2 (faz API calls para ECS, envia logs)
  • Task IAM Role: role específica por task (acesso ao S3, DynamoDB, etc.)

ECS — ALB Integration e Dynamic Port Mapping

Permite múltiplas instâncias do mesmo container na mesma EC2. O ALB descobre a porta automaticamente. Aumenta resiliência e utilização de CPU.

Estratégia de Resolução

  1. Leia a última frase primeiro — geralmente contém o que está sendo pedido (menor custo? menor esforço operacional? menor RTO?)
  2. Identifique os requisitos — grife mentalmente: "multi-region", "compliance", "menor latência", etc.
  3. Elimine opções absurdas — alternativas com serviços em contexto errado (ex: Glacier para dados acessados frequentemente)
  4. Entre 2 opções válidas, escolha a mais AWS-native — managed services > self-managed
  5. Menor esforço operacional = serverless, managed services, automação
⏱️

Gestão de Tempo

  • 75 questões em 180 minutos = 2.4 minutos por questão
  • Questões longas (cenário grande): marque e volte depois
  • Primeiro passe: responda tudo que sabe com confiança
  • Segundo passe: revise as marcadas com mais calma
  • NÃO gaste mais de 4 minutos em uma questão na primeira passada
🧠

Padrões de Resposta Comuns

  • "Menor esforço operacional" → Prefira serviços gerenciados (Aurora > RDS MySQL > MySQL on EC2)
  • "Menor custo" → Cuidado: pode ser custo imediato OU TCO. Serverless geralmente vence em TCO.
  • "Tempo de migração mais curto" → Rehost > Replatform > Refactor
  • "Mínimo downtime na migração" → DMS com CDC, Blue/Green deploy
  • "Compliance / regulatório" → KMS CMK, CloudTrail, AWS Config, SCPs
  • "Multi-region active-active" → DynamoDB Global Tables, Aurora Global DB
  • "Loosely coupled" → SQS, SNS, EventBridge, Step Functions
